Moongrow is a Turborepo monorepo with four distinct layers: three Next.js frontends, an Express API backend, a PostgreSQL database, and an Anchor smart contract on Solana.

## Monorepo structure

```
moongrow/
├── apps/
│   ├── retailer/     # coffeeshop owner dashboard
│   ├── budtender/    # staff mobile PWA
│   └── admin/        # Moongrow internal ops
├── packages/
│   ├── api/          # Express server — POS, campaigns, payout queue
│   ├── db/           # Prisma schema + client
│   └── shared/       # types, Zod validators, constants
└── programs/
    └── moongrow/     # Anchor (Rust) — vault + payout instructions
```

**Package names:** `@moongrow/retailer`, `@moongrow/budtender`, `@moongrow/admin`, `@moongrow/api`, `@moongrow/db`, `@moongrow/shared`

## Key design principles

**Non-custodial vaults.** Each store's EURC is held in a PDA controlled by the Anchor program, not Moongrow's wallet. Retailers can withdraw at any time.

**Immutable sale log.** `SaleEvent` records are append-only. Campaign rates are snapshotted at ingestion — edits to campaigns never affect in-flight sales.

**Physical double-pay prevention.** The `PayoutRecord` PDA is seeded by `sale_event_id`. A second `execute_payout` call for the same sale fails with a Solana account collision error before any logic runs.

**Queue-based resilience.** BullMQ handles retries with exponential backoff. If a vault is empty, the payout job retries automatically when funds are added.
